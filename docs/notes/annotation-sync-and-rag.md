# Annotation Sync And RAG

This document explains how Notes PDF annotation state is saved on the server,
how it is synchronized with `.codmes` files, and how saved PDF notes enter
search/RAG.

## Main Files

- API routes: `server/index.mjs`
- Annotation storage paths and document ingest: `server/lib/document-ingest.mjs`
- Search/RAG service: `server/lib/search-service.mjs`
- Notes tool mode: `server/lib/runtime/tool-mode-registry.mjs`
- Workspace tool definitions: `server/lib/runtime/workspace-tools.mjs`
- Apple API client: `client/apple/Sources/Codmes/WorkspaceAPI.swift`
- Apple note models: `client/apple/Sources/Codmes/Models.swift`

## Annotation API

The Apple client uses:

```text
GET /api/file/annotations?path=Notes/mypage.pdf
PUT /api/file/annotations?path=Notes/mypage.pdf
```

The Swift client methods are:

```swift
fileAnnotations(path:)
saveFileAnnotations(path:annotations:)
```

On `GET`, the server:

1. Resolves the requested workspace-relative path.
2. Rejects folders.
3. Reads the primary annotation JSON path.
4. If missing, tries to migrate older annotation state locations.
5. If no state exists, returns an empty schema version 2 annotation document.

On `PUT`, the server:

1. Resolves the requested workspace-relative path.
2. Rejects folders.
3. Normalizes the body with `schemaVersion >= 2`, the current `documentPath`,
   a fresh `updatedAt`, `pages`, `objects`, and `elements`.
4. Writes pretty JSON to the primary annotation state path.
5. Refreshes the search index for the PDF path.
6. Returns the normalized annotation document to the client.

## Annotation Storage Location

For:

```text
Notes/mypage.pdf
```

the primary editable annotation state is:

```text
Notes/.codmes/annotations/mypage.codmes.json
```

This is generated by `annotationsPathForDocument(workspaceRoot, relativePath)`.
The state is placed beside the PDF folder so moving or copying a document folder
can carry its editable note state.

The server also recognizes older paths:

- Content-scoped path under the top-level content root, encoded by file path.
- Legacy workspace-global `.codmes/annotations/<encoded>.json`.

When reading, `readAnnotationsForDocument(...)` and `migrateLegacyAnnotations`
copy older state into the primary path when possible.

When moving, copying, or deleting files/folders, `server/index.mjs` also moves,
copies, or removes matching document state files so PDF notes follow their PDF.

## Annotation Data Model

The server persists `PDFAnnotationDocument`:

```text
PDFAnnotationDocument
├── schemaVersion
├── documentPath
├── updatedAt
├── pages[]
├── objects[]
└── elements[]
```

Important page fields:

- `pageIndex`: zero-based PDF page.
- `inkDataBase64`: legacy PencilKit drawing data.
- `inkStrokes`: portable Codmes ink strokes.
- `objects`: page-local text/image objects.
- `elements`: canonical element-model entries for strokes, shapes, text, and
  images.

Important object fields:

- `id`: stable object id.
- `type`: usually `text` or `image`.
- `pageIndex`: zero-based PDF page.
- `bbox`: normalized page rectangle.
- `text`: text-box content or OCR text.
- `dataBase64`: embedded image payload.
- `metadata`: font size, color, MIME type, file name, OCR metadata, and other
  UI hints.

The Apple model still supports legacy `inkStrokes` and `objects`, but
`syncNoteElementsFromLegacy()` keeps the newer `elements` model in sync.

## Client Save Flow

1. User draws, erases, moves lasso content, edits text/images, imports pages,
   or uses undo/redo.
2. The client updates local `annotations`.
3. `scheduleSave(...)` waits briefly so rapid edits do not spam the server.
4. `saveAnnotations(...)` sends `PUT /api/file/annotations`.
5. The server writes `.codmes/annotations/*.codmes.json`.
6. The server refreshes search for the PDF.
7. The returned saved document replaces local state.

The server stores the current annotation document only. Client-only UI state,
such as current handles, live previews, and undo/redo history, is not persisted
as server history.

## Search And RAG Indexing

Search/RAG is workspace-wide, not Notes-only. Notes participates by saving
annotation state that the document ingest layer can read.

The relevant index paths are:

```text
.codmes/index/search.json
.codmes/index/documents/*.json
.codmes/index/annotation-ocr/*.json
```

When annotation JSON is saved, `writeFileAnnotations(...)` calls:

```text
refreshSearchIndexPaths([relativePath])
```

That flows into `updateSearchIndex(...)` in `server/lib/search-service.mjs`.
The search index can then include:

- PDF source text extracted from the PDF itself.
- Text annotation objects as `source: "annotation-text"`.
- Image annotation OCR blocks as `source: "annotation-image-ocr"`.
- Regular Markdown/text/document chunks from the workspace.
- Conversation and memory context through the runtime search tools.

Handwritten `inkStrokes` are not converted to text today, so they are not
searchable unless a future handwriting OCR layer adds text blocks.

## Annotation OCR

`extractDocumentAnnotationBlocks(...)` reads saved annotation JSON for a PDF and
collects searchable annotation blocks.

Text objects:

- If `object.type` is text-like and `object.text` is non-empty, it emits an
  `annotation-text` block.

Image objects:

- If `object.text` or `object.metadata.ocrText` exists, it emits an
  `annotation-image-ocr` block immediately.
- Otherwise, if VLM OCR is enabled and `dataBase64` exists, the server sends the
  image payload to the configured VLM OCR path.
- OCR output is cached by content hash under
  `.codmes/index/annotation-ocr/`.

OCR is opportunistic. If annotation image OCR fails, the base PDF text remains
searchable and indexing continues.

## Notes Chat Tool Mode

The Notes surface default tool mode is defined in
`server/lib/runtime/tool-mode-registry.mjs`.

Enabled by default:

- `workspace_search`
- `codmes_search`
- `read_note_file`
- `read_file_metadata`
- `conversation_search`
- `memory_search`
- `tool_discovery`
- `conversation_read`

Not enabled by default in Notes:

- `apply_patch`
- `run_checks`
- `run_git_command`
- other Code-surface mutation tools

This means a Notes conversation can search and read note context, but it does
not automatically receive code-edit privileges.

`tool_discovery` can recommend a temporary expansion when a turn clearly needs
a specialized capability, but the normal Notes posture is read/search first.

## PDF Export And `.codmes` State

The Apple client supports two export styles:

- Flattened annotated PDF: renders PDF pages plus current annotation state into
  a new PDF.
- Codmes state export: writes the PDF plus a `.codmes.json` annotation state
  file so editable notes can be re-imported.

The server-side persisted `.codmes/annotations/*.codmes.json` remains the
source of truth for editable state in the workspace. Flattened PDFs are export
artifacts; they are useful for sharing, but they are not the primary editable
model.

## What Is Not Implemented Yet

- Handwriting OCR over pen `inkStrokes`.
- Windows and Android/Galaxy Tab render/edit adapters.
- True PDF-standard annotation round-trip as the primary editing model.
- Vector database semantic ranking beyond the current Codmes search index path.
